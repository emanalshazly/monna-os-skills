# Skill Authoring Guide

> **"A great skill is not a great prompt. A great skill is a great system."**

## The SKILL.md Spec

All skills in this collection follow the [Anthropic Agent Skills specification](https://agentskills.io) with MONNA extensions for quality governance.

### Required Frontmatter

```yaml
---
name: <kebab-case-skill-name>          # Required. Unique within the collection.
version: semver                         # Required. Start at 1.0.0.
description: >                          # Required. <1024 chars. Include "Use when..."
  Use when the user needs to [specific task].
  [What it does]. [What it does NOT do].
triggers:                               # Required. 3–7 trigger phrases.
  - "trigger phrase 1"
  - "trigger phrase 2"
categories:                             # Required. From our taxonomy.
  - governance
  - compliance
quality_tier: draft                     # Required. Will be updated by CI.
compatibility:                          # Required. Tested runtimes.
  kimi: unknown
  claude: unknown
  copilot: unknown
  cursor: unknown
  openclaw: unknown
fingerprint: "<domain>-###-<hash>"      # Required. Generated by overlap checker.
last_tested: YYYY-MM-DD                 # Required. Updated by CI.
author: <github-username>               # Required.
license: MIT                            # Required.
---
```

### MONNA Extensions (Required for this collection)

| Field | Purpose |
|-------|---------|
| `quality_tier` | Draft / validated / reviewed / certified / universal |
| `compatibility` | Runtime-by-runtime certification status |
| `fingerprint` | Semantic fingerprint for overlap detection |
| `last_tested` | Date of last compatibility test |

## Writing Great Skills

### 1. Trigger Precision

Bad: `"help with AI"`  
Good: `"design an AI governance framework aligned with NIST RMF and EU AI Act"`

Triggers are how agents decide to activate your skill. Vague triggers = false activation = wasted context.

### 2. Scope Honesty

Every skill MUST have a **"When NOT to Use This Skill"** section. This prevents the skill from overreaching and giving bad advice outside its domain.

Example:
```markdown
## When NOT to Use This Skill

- This skill does NOT provide legal advice. Consult a qualified attorney for regulatory interpretation.
- This skill does NOT replace formal risk assessment by certified professionals.
- This skill is NOT a substitute for organizational policy review.
```

### 3. Evidence-Based Content

Every significant claim in a skill should be traceable:
- Cite standards (NIST SP 800-37, EU AI Act Article 9, ISO 42001 Clause 5.1)
- Cite sources (arXiv papers, official documentation, recognized frameworks)
- Use passage-level traceability: "Per NIST RMF Section 3.2..."

### 4. Progressive Disclosure

Structure your skill so agents load the right depth:
- **Frontmatter** — always loaded (metadata, triggers)
- **SKILL.md body** — loaded on activation (core principles, step-by-step)
- **references/** — loaded on demand (deep source materials, standards mappings)
- **examples/** — loaded on demand (worked examples)

### 5. Anti-Patterns

The anti-patterns section is one of the most valuable parts of a skill. It tells the agent what NOT to do.

Example:
```markdown
## Anti-Patterns

- Do NOT treat the risk assessment as a one-time exercise. It must be continuous.
- Do NOT conflate "low risk" with "no risk." The EU AI Act's minimal risk category still requires documentation.
- Do NOT skip stakeholder engagement. Governance frameworks fail when imposed top-down.
```

### 6. Worked Examples

Every skill aiming for REVIEWED tier or higher must include at least one worked example:
- Realistic input scenario
- Step-by-step application of the skill
- Expected output or deliverable format

## Domain Taxonomy

Use these categories in your skill frontmatter:

- `governance` — AI governance, compliance, policy
- `security` — threat modeling, red-teaming, risk assessment
- `data-governance` — data quality, lineage, stewardship
- `infrastructure` — DePIN, node operations, cloud infrastructure
- `multilingual` — translation, localization, cross-cultural communication
- `tooling` — skill validation, overlap detection, quality scoring

## Validation Before Submitting

```bash
# 1. Check structure
python tools/validator/validate.py skills/<domain>/<skill-name>/

# 2. Check overlap
python tools/overlap-checker/check.py skills/<domain>/<skill-name>/SKILL.md

# 3. Self-review against this guide
# 4. Submit PR
```

## Common Mistakes

1. **Vague triggers** — causes false activation
2. **Missing scope boundaries** — skill overreaches
3. **No references** — claims are unsubstantiated
4. **No examples** — users can't see how to apply it
5. **Copy-paste from commercial skills** — this repo is open-source, distinct from commercial portfolio
6. **Ignoring the "When NOT to Use" section** — scope honesty failure
